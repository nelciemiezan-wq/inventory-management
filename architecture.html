Data Flow — A Single Filtered Request

From a filter change in the UI to a re-rendered chart, the same path applies for every read in the app.

  1. User changes a filter Warehouse, Category, Order Status, or Time Period in FilterBar.vue.
  2. Composable updates shared state useFilters.js mutates the relevant ref; consumers re-run.
  3. View triggers an API call A view (e.g. Dashboard.vue) calls api.getDashboardSummary(filters); src/api.js serializes filters into query params, skipping any equal to 'all'.
  4. FastAPI receives the request Route handler in server/main.py accepts optional query params, then runs apply_filters() and filter_by_month() over the in-memory list.
  5. Pydantic validates the response The handler returns typed objects (InventoryItem, Order, …); FastAPI serializes them to JSON.
  6. Vue stores raw data in a ref The view assigns the response to e.g. allOrders.value; computed properties derive totals, groupings, and chart series.
  7. Template re-renders Vue's reactivity pushes the new computed values into the DOM and SVG charts.

Key Conventions

Filter pipeline

Four filters — Time Period, Warehouse, Category, Order Status — flow uniformly: Vue composable → query params → server-side list comprehension. A value of all is treated as "skip this filter".

Reactivity model

Views hold raw API data in ref()s (e.g. allOrders, inventoryItems); everything displayed — totals, groupings, chart series — is a computed() derived from those refs.

Time dimension

Months use YYYY-MM; quarters use Q1-2025 … Q4-2025 and expand to their three months server-side via QUARTER_MAP. Inventory has no time dimension, so the month filter is ignored there.

Persistence

All writes (tasks, purchase orders) mutate in-memory lists only. Restarting the server reloads JSON from server/data/ and discards changes — appropriate for a demo, not production.
